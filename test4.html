III. Benchmark Methodology
3.2 Evaluation Criteria and Dimensions
Evaluation Criteria and Benchmark Dimensions
The benchmark is multi-dimensional and not limited to a single metric
The benchmark work covers several analysis axes. Footprint analysis and execution-time analysis belong to the broader functional analysis, while static analysis addresses source-code structure and SDK composition.
F

Functional Analysis

Feature-level comparison and implementation behavior

  • Compares how the same functionality is implemented across vendors
  • Focuses on scenario execution under equivalent observable behavior
  • Includes both footprint and execution-time perspectives
M

Footprint Analysis

Part of the functional analysis

  • Code size
  • Memory occupation
  • Per-layer and per-module contribution to the global footprint
T

Execution-Time Analysis

Part of the functional analysis

  • Measured efficiency in selected benchmark scenarios
  • Initialization and runtime behavior
  • Performance differences under controlled conditions
S

Static Analysis Orientation

Code structure and ecosystem quality view

  • Code structure and composition
  • Dependency behavior
  • API surface, documentation, duplication, complexity, compliance, and coverage
Important note: the exact scope depends on the benchmark campaign, and not all dimensions were at the same maturity level during the internship.
Key message: the benchmark is multi-dimensional: it combines feature-level behavior, footprint, execution efficiency, and software-structure analysis.
