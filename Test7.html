USB Benchmark - Enhancement Proposals
Enhancement Proposals from the USB Benchmark
Proposals derived from the ST vs GD32 USB analysis to reduce ROM overhead while preserving STM32Cube strengths
R

Main ROM Gap

+77% to +88%

STM32 ROM footprint is significantly higher than GD32 in HID and CDC scenarios.

M

Main Strength

-57% to -65%

STM32 uses less RAM and achieves much faster USB initialization.

C

Root Cause

HAL Layer

HAL RCC and HAL/LL USB driver explain most of the ROM difference.

Diagnosis

Main Contributors

  • HAL RCC contributes about 4560 bytes
  • HAL/LL USB driver contributes about 5006 bytes
  • Externalized USB configuration files increase the application footprint
Interpretation

Why the Gap Exists

STM32Cube uses a more modular architecture: HAL, middleware, and user configuration are separated. This improves flexibility and reuse, but it comes with a ROM cost.

Objective

Improvement Direction

Reduce ROM overhead without losing the STM32 advantages: modularity, lower RAM usage, and faster initialization.

1

Optimize HAL RCC

  • Trim the clock-configuration logic to keep only the paths needed by the USB scenario
  • Improve conditional compilation or dead-code elimination in RCC-related code
High Priority
↓
2

Refine USB Driver Feature Selection

  • Exclude unused USB code paths from Full-Speed-only projects
  • Separate broader USB capabilities from minimal scenario-specific builds
High Priority

3

Review Configuration File Overhead

  • Limit the footprint introduced by user configuration files
  • Keep flexibility while reducing unnecessary compiled code

  • Lower STM32 USB ROM footprint

Engineering logic: the benchmark not only shows that STM32 ROM is higher, it also identifies where the gap comes from, which makes the proposals precise and actionable.
Main takeaway: the most impactful improvements for the USB benchmark are HAL RCC optimization and finer compile-time USB feature selection.
